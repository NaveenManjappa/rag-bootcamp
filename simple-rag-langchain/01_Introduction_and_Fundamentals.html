1. What is LangChain? 🤔

🔰 BEGINNER SECTION

LangChain is an open-source framework that makes it easy to build applications powered by Large Language Models (LLMs) like GPT-4, Claude, or Llama.

Why LangChain?

Imagine you want to build a chatbot that can:

  • Answer questions about your company's PDF documents
  • Search through your database
  • Remember previous conversations
  • Call external APIs

Without LangChain, you'd need to:

  1. ✍️ Write code to load and parse PDFs
  2. ✍️ Convert text to embeddings
  3. ✍️ Store embeddings in a vector database
  4. ✍️ Implement semantic search
  5. ✍️ Format prompts for the LLM
  6. ✍️ Handle LLM API calls
  7. ✍️ Manage conversation memory

With LangChain:

  • ✅ All these components are pre-built and ready to use
  • ✅ You just connect them like LEGO blocks
  • ✅ Focus on your application logic, not infrastructure

🎓 INTERMEDIATE NOTE

LangChain provides:

  • Abstractions: Unified interfaces for different LLMs, vector stores, and tools
  • Chains: Composable workflows using LCEL (LangChain Expression Language)
  • Agents: Autonomous systems that can use tools and make decisions
  • Memory: Conversation history and context management
  • Callbacks: Monitoring, logging, and debugging hooks

4. Core Concepts 📚

4.1 Documents 📄

🔰 BEGINNER

A Document is LangChain's way of representing a piece of text with metadata.

Think of it like a note card:

  • Front (page_content): The actual text
  • Back (metadata): Information about the text (source, page number, date, etc.)
Python 
# Creating a Document manually
from langchain_core.documents import Document

# Create a simple document
doc = Document(
    page_content="LangChain makes building LLM applications easy!",
    metadata={
        "source": "introduction.txt",
        "author": "LangChain Team",
        "date": "2025-01-15"
    }
)

# Access the content
print("Content:")
print(doc.page_content)
print("\nMetadata:")
print(doc.metadata)
print(f"\nSource: {doc.metadata['source']}")

🎓 INTERMEDIATE: Why Metadata Matters

Metadata is crucial for:

  1. Citation: Showing users where information came from
  2. Filtering: Only search documents from specific sources
  3. Debugging: Tracking which documents are being retrieved
  4. Analytics: Understanding which sources are most useful
Python 
# Creating multiple documents (like a mini knowledge base)
documents = [
    Document(
        page_content="Python is a high-level programming language.",
        metadata={"category": "programming", "difficulty": "beginner"}
    ),
    Document(
        page_content="Machine learning is a subset of artificial intelligence.",
        metadata={"category": "AI", "difficulty": "intermediate"}
    ),
    Document(
        page_content="RAG combines retrieval and generation for better LLM outputs.",
        metadata={"category": "AI", "difficulty": "advanced"}
    )
]

# Print all documents
for i, doc in enumerate(documents, 1):
    print(f"\n📄 Document {i}:")
    print(f"   Content: {doc.page_content}")
    print(f"   Category: {doc.metadata['category']}")
    print(f"   Difficulty: {doc.metadata['difficulty']}")

4.2 LCEL (LangChain Expression Language) 🔗

🔰 BEGINNER

LCEL is a way to connect different components using the pipe operator |.

Think of it like a factory assembly line:

Diagram 
Input → Component 1 → Component 2 → Component 3 → Output

Before LCEL (Old Way - Messy!):

Python 
output = component3(component2(component1(input)))

With LCEL (New Way - Clean!):

Python 
chain = component1 | component2 | component3
output = chain.invoke(input)

🎓 INTERMEDIATE: LCEL Deep Dive

LCEL provides:

  • Streaming: Stream outputs as they're generated
  • Batch Processing: Process multiple inputs efficiently
  • Async Support: Non-blocking operations
  • Debugging: Better error messages and logging
  • Type Safety: Better IDE autocomplete

Key Methods:

  • .invoke(input): Process single input
  • .batch([input1, input2]): Process multiple inputs
  • .stream(input): Stream output tokens
  • .ainvoke(input): Async version of invoke
Python 
# Simple LCEL Example: String Transformation Chain
from langchain_core.runnables import RunnableLambda

# Create simple transformation functions
def uppercase(text: str) -> str:
    """Convert text to uppercase"""
    print(f"  Step 1: uppercase → {text.upper()}")
    return text.upper()

def add_prefix(text: str) -> str:
    """Add a prefix to text"""
    result = f"RESULT: {text}"
    print(f"  Step 2: add_prefix → {result}")
    return result

def add_emoji(text: str) -> str:
    """Add emoji to text"""
    result = f"✅ {text}"
    print(f"  Step 3: add_emoji → {result}")
    return result

# Create runnables (components that can be chained)
uppercase_runnable = RunnableLambda(uppercase)
prefix_runnable = RunnableLambda(add_prefix)
emoji_runnable = RunnableLambda(add_emoji)

# Build the chain using LCEL
chain = uppercase_runnable | prefix_runnable | emoji_runnable

# Execute the chain
print("Input: 'hello langchain'")
print("\nProcessing:")
result = chain.invoke("hello langchain")
print(f"\nFinal Output: {result}")

5. Quick Start Example 🚀

🔰 BEGINNER: Your First LLM Call

Let's make our first call to an LLM using LangChain!

Python 
# Import ChatOpenAI (the LLM interface)
from langchain_openai import ChatOpenAI

# Initialize the LLM
# model: Which GPT model to use
# temperature: 0 = deterministic, 1 = creative
llm = ChatOpenAI(

# Make a simple call
response = llm.invoke("What is LangChain in one sentence?")

# Print the response
print("Question: What is LangChain in one sentence?")
print(f"\nAnswer: {response.content}")

🔰 BEGINNER: Using Prompts

Instead of plain strings, use prompt templates for better control:

from langchain_core.prompts import ChatPromptTemplate

# Create a prompt template
# {topic} is a variable we'll fill in later
prompt = ChatPromptTemplate.from_template(
    "Explain {topic} in simple terms suitable for beginners."

# View the prompt structure
print("Prompt Template:")

Python 
# Build a chain: Prompt → LLM
from langchain_core.output_parsers import StrOutputParser

# Components:
# 1. prompt: Formats the input
# 2. llm: Generates the response
# 3. StrOutputParser: Extracts just the text from the response

chain = prompt | llm | StrOutputParser()

# Use the chain with different topics
topics = ["machine learning", "embeddings", "vector databases"]

for topic in topics:
    print(f"\n{'='*60}")
    print(f"Topic: {topic.upper()}")
    print('='*60)

    # Invoke the chain
    explanation = chain.invoke({"topic": topic})
    print(explanation)

🎓 INTERMEDIATE: Batch Processing

Process multiple inputs efficiently using .batch():

Python 
# Batch process multiple topics at once
topics_batch = [
    {"topic": "RAG"},
    {"topic": "LCEL"},
    {"topic": "LangChain agents"}
]

# Batch processing is more efficient than calling invoke() multiple times
results = chain.batch(topics_batch)

# Print results

Example: Building a Q&A System

Without LangChain (50+ lines):

Python 
import openai
import PyPDF2
import faiss
import numpy as np

# 1. Load PDF manually
def load_pdf(file_path):
    reader = PyPDF2.PdfReader(file_path)
    text = ""
    for page in reader.pages:
        text += page.extract_text()
    return text

# 2. Split text manually
def split_text(text, chunk_size=1000):
    chunks = []
    for i in range(0, len(text), chunk_size):
        chunks.append(text[i:i+chunk_size])
    return chunks

# 3. Create embeddings manually
def create_embeddings(chunks):
    embeddings = []
    for chunk in chunks:
        response = openai.Embedding.create(
            input=chunk,
            model="text-embedding-ada-002"
        )
        embeddings.append(response['data'][0]['embedding'])
    return np.array(embeddings)

# 4. Create vector store manually
def create_vector_store(embeddings):
    dimension = embeddings.shape[1]
    index = faiss.IndexFlatL2(dimension)
    index.add(embeddings)
    return index

# 5. Search manually
def search(query, index, chunks):
    query_embedding = openai.Embedding.create(
        input=query,
        model="text-embedding-ada-002"
    )['data'][0]['embedding']

    distances, indices = index.search(
        np.array([query_embedding]), k=3
    )
    return [chunks[i] for i in indices[0]]

# 6. Generate answer manually
def generate_answer(query, context):
    prompt = f"Context: {context}\n\nQuestion: {query}\n\nAnswer:"
    response = openai.ChatCompletion.create(
        model="gpt-3.5-turbo",
        messages=[{"role": "user", "content": prompt}]
    )
    return response['choices'][0]['message']['content']

# Use it:
text = load_pdf("document.pdf")
chunks = split_text(text)
embeddings = create_embeddings(chunks)
index = create_vector_store(embeddings)
relevant_chunks = search("What is RAG?", index, chunks)
answer = generate_answer("What is RAG?", " ".join(relevant_chunks))

With LangChain (10 lines!):

Python 
from langchain_community.document_loaders import PyPDFLoader
from langchain_text_splitters import RecursiveCharacterTextSplitter
from langchain_openai import OpenAIEmbeddings, ChatOpenAI
from langchain_community.vectorstores import FAISS
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough

# Load, split, embed, and store
docs = PyPDFLoader("document.pdf").load()
chunks = RecursiveCharacterTextSplitter(chunk_size=1000).split_documents(docs)
vectorstore = FAISS.from_documents(chunks, OpenAIEmbeddings())
retriever = vectorstore.as_retriever()

# Build RAG chain
prompt = ChatPromptTemplate.from_template(
    "Context: {context}\n\nQuestion: {question}\n\nAnswer:"
)
chain = (
    {"context": retriever, "question": RunnablePassthrough()}
    | prompt | ChatOpenAI() | StrOutputParser()
)

# Use it:
answer = chain.invoke("What is RAG?")

⚡ Quick Recap

  • LangChain is a framework that provides pre-built, composable components (loaders, splitters, embedding models, vector stores, retrievers, prompts, LLMs) so you don't build LLM application infrastructure from scratch.
  • As of 1.0+, LangChain is split into focused packages — langchain-core, langchain-community, langchain-openai, langchain-text-splitters — and you import from the specific package you need rather than a single monolithic langchain package.
  • A Document is the core data unit: it pairs page_content (the raw text) with metadata (a dict used for citation, filtering, debugging, and analytics).
  • LCEL (LangChain Expression Language) uses the | pipe operator to compose components into a chain, replacing deeply nested function calls with a linear, readable pipeline.
  • Runnables built with LCEL expose a consistent set of methods: .invoke() for one input, .batch() for many inputs at once, .stream() for token-by-token output, and .ainvoke() for async execution.
  • A typical chain pattern is Prompt → LLM → OutputParser, where the prompt template fills in variables, the LLM generates a response, and StrOutputParser extracts plain text from the response object.
  • An LLM response is an AIMessage object, not just a string — it carries .content plus response_metadata such as token usage and model name, which is useful for cost tracking and debugging.
  • Compared to hand-rolled pipelines (manual PDF parsing, embedding calls, FAISS indexing, and prompt formatting), LangChain collapses the same RAG workflow from 50+ lines down to roughly 10 by relying on its built-in loaders, splitters, vector store integrations, and LCEL chains.

🎯 Interview Prep

1. What problem does LangChain actually solve, and when would you NOT use it?

LangChain solves the integration and orchestration problem in LLM app development — it gives you standard interfaces for document loading, chunking, embeddings, vector stores, and prompt/LLM composition so you're not hand-writing glue code for every provider. I'd reach for it when I need to combine multiple components (retrieval + generation, multi-step chains, agents) or swap providers without rewriting logic. I'd avoid it for a single, simple API call to one LLM provider where the abstraction overhead isn't paying for itself, or in latency/dependency-sensitive services where a thinner custom client is easier to reason about.

Gotcha: An interviewer may push on "abstraction cost" — be ready to discuss debugging harder-to-trace chains versus raw SDK calls.

2. Explain LCEL and why the pipe operator matters.

LCEL, LangChain Expression Language, lets you compose Runnables with the | operator so a chain reads left-to-right as data flow, e.g. prompt | llm | StrOutputParser(), instead of nested function calls like parser(llm(prompt(input))). Every LCEL component implements a common Runnable interface, which is what gives you .invoke(), .batch(), .stream(), and .ainvoke() "for free" on any chain, no matter how it's composed. That uniform interface is also what enables cross-cutting features like streaming and async to work consistently across arbitrarily composed chains.

Gotcha: Be ready to explain what actually happens under the hood when two Runnables are piped — it constructs a RunnableSequence.

3. What is a Document object and why not just pass around raw strings?

A Document pairs page_content with a metadata dictionary, so a chunk of text always travels with context like its source, page number, author, or category. Raw strings lose that provenance, which matters for citing sources back to the user, filtering retrieval by metadata (e.g., only search a specific category), and debugging which source produced a bad answer. In a RAG pipeline specifically, metadata is what lets you show "this answer came from page 12 of document X" rather than an opaque blob of text.

Gotcha: They might ask how metadata is used at retrieval time — mention metadata filtering supported by most vector stores, not just post-hoc display.

4. Why did LangChain split into langchain-core, langchain-community, langchain-openai, etc. in 1.0+?

Splitting into focused packages reduces the dependency footprint — you no longer pull in every integration's SDK just to use core abstractions, which was a real pain point with the old monolithic package. It also improves modularity and versioning: langchain-openai can ship a fix without forcing a release of unrelated community integrations. From a consumer's perspective, it means faster installs, fewer version conflicts, and clearer signals about which package owns which functionality.

Gotcha: Know at least one concrete import mapping, e.g. Document comes from langchain_core.documents, not langchain.

5. What's the difference between .invoke(), .batch(), and .stream() on a chain?

.invoke() runs the chain once on a single input and blocks until the full result is ready. .batch() takes a list of inputs and processes them more efficiently than looping .invoke() calls, since the underlying implementation can parallelize or otherwise optimize multiple requests. .stream() returns output incrementally as it's generated, which matters for user-facing applications like chat UIs where you want tokens to appear as they're produced rather than waiting for the entire response.

Gotcha: They may probe on the async variants (.ainvoke(), .abatch(), .astream()) and when you'd choose async over sync (I/O-bound, high-concurrency services).

6. Walk through what response.response_metadata gives you and why it matters in production.

The response from an LLM call is an AIMessage, and its response_metadata includes things like token_usage (prompt, completion, and total tokens), the exact model_name used, and the finish_reason. In production this is essential for cost tracking, since token counts drive billing, and for debugging — finish_reason tells you if a response was truncated versus completed normally. It also lets you build usage dashboards or per-user rate limiting without instrumenting the LLM call yourself.

Gotcha: They might ask how you'd aggregate token usage across a .batch() call or a multi-step chain — you'd need to sum metadata across each individual LLM invocation, since it's per-call, not per-chain.

7. How would you explain the difference between LangChain and building the same pipeline by hand, beyond "it's less code"?

Beyond line count, LangChain gives you tested, maintained integrations for each stage of the pipeline — PDF loading, chunking, embeddings, vector storage — so you're not maintaining bespoke wrappers around every provider's SDK quirks. It also gives you composability for free: because every stage implements the same Runnable interface, swapping FAISS for another vector store or one embedding model for another is a one-line change instead of a rewrite. The tradeoff is you're depending on LangChain's abstractions keeping pace with each underlying provider's API changes, and debugging failures sometimes means understanding an extra abstraction layer.

Gotcha: Be prepared to name a concrete downside, like abstraction leakage or version churn, rather than presenting LangChain as strictly superior with no cost.

8. What are Chains, Agents, and Memory, and how do they relate to each other?

A Chain is a fixed sequence of steps, like Prompt → LLM → OutputParser, where the flow of execution is defined ahead of time by the developer. An Agent is a more autonomous system that uses an LLM to decide, at runtime, which tools to call and in what order, rather than following a hardcoded sequence. Memory sits alongside both, storing conversation history or other state so that a chain or agent can maintain context across multiple turns instead of treating each call as stateless.

Gotcha: They may ask you to draw the line between "a chain with conditional logic" and "an agent" — the key distinction is whether the LLM itself is choosing the next action versus following developer-defined control flow.
