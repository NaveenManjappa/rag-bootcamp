1. Why Document Loaders? 🤔

🔰 BEGINNER

Document Loaders are tools that help you convert files (PDFs, CSVs, web pages, etc.) into Document objects that LangChain can work with.

Think of them as translators:

  • Input: Files in various formats (PDF, CSV, JSON, HTML)
  • Output: Standardized Document objects with text content and metadata

Why is this important?

Every RAG application needs to:

  1. 📥 Load data from various sources
  2. 🔄 Convert it to a standard format
  3. 📊 Extract metadata (source, page number, etc.)
  4. 🎯 Prepare it for embedding and retrieval

Document Loaders handle all of this automatically!

🎓 INTERMEDIATE

All document loaders in LangChain implement the same interface:

  • .load(): Load all documents at once (returns list[Document])
  • .lazy_load(): Load documents one at a time (generator, memory efficient)

This consistency makes it easy to switch between different data sources.

# Verify setup

2. Document Object Structure 📄

🔰 BEGINNER

Every Document has two main parts:

  1. page_content: The actual text (string)
  2. metadata: Information about the document (dictionary)

Think of it like a book:

  • page_content = The story
  • metadata = The cover information (title, author, page number, etc.)
Python 
from langchain_core.documents import Document

# Create a sample document
doc = Document(
    page_content="This is the actual content of the document. It contains the text we want to process.",
    metadata={
        "source": "example.pdf",
        "page": 1,
        "author": "John Doe",
        "date": "2025-01-15"
    }
)

# Inspect the document
print("📄 Document Structure:")
print(f"\nType: {type(doc)}")
print(f"\nContent (first 100 chars): {doc.page_content[:100]}...")
print(f"\nMetadata: {doc.metadata}")
print(f"\nSource: {doc.metadata['source']}")
print(f"Page Number: {doc.metadata['page']}")

3. Loading PDF Files 📕

🔰 BEGINNER

PyPDFLoader is used to load PDF files. It:

  • Extracts text from each page
  • Creates one Document per page
  • Automatically adds source and page number to metadata

Example 1: Loading a Single PDF

Python 
from langchain_community.document_loaders import PyPDFLoader

# Load the "Attention is All You Need" paper (if it exists)
pdf_path = "attention.pdf"

if Path(pdf_path).exists():
    print(f"Loading PDF: {pdf_path}")
    print("⏳ This may take a moment...\n")

    # Create loader
    loader = PyPDFLoader(pdf_path)

    # Load all pages
    documents = loader.load()

    print(f"✅ Loaded {len(documents)} pages\n")

    # Inspect first page
    print("📄 First Page:")
    print(f"   Content (first 200 chars): {documents[0].page_content[:200]}...")
    print(f"\n   Metadata: {documents[0].metadata}")

    # Inspect last page
    print(f"\n📄 Last Page (page {len(documents)}):")
    print(f"   Content (first 200 chars): {documents[-1].page_content[:200]}...")

else:
    print(f"❌ PDF not found: {pdf_path}")
    print("   Make sure the file exists in the project root")

🎓 INTERMEDIATE: Lazy Loading for Large PDFs

For very large PDFs, use .lazy_load() to process one page at a time:

Python 
# Lazy loading example
if Path(pdf_path).exists():
    loader = PyPDFLoader(pdf_path)

    print("🔄 Lazy loading pages (memory efficient):")

    # Process first 3 pages only
    for i, page in enumerate(loader.lazy_load()):
        if i >= 3:  # Only process first 3 pages for demo
            break

        print(f"\nPage {i+1}:")
        print(f"  Length: {len(page.page_content)} characters")
        print(f"  Preview: {page.page_content[:100]}...")

    print("\n💡 Tip: Use lazy_load() for PDFs > 100 pages to save memory")

Example 2: Loading Multiple PDFs from a Directory

Python 
# Load all PDFs from the pdfs/ directory
pdf_directory = "pdfs"

if Path(pdf_directory).exists():
    print(f"📂 Loading PDFs from: {pdf_directory}/\n")

    all_documents = []

    # Find all PDF files
    pdf_files = list(Path(pdf_directory).glob("*.pdf"))
    print(f"Found {len(pdf_files)} PDF files:")

    for pdf_file in pdf_files:
        print(f"  - {pdf_file.name}")

        # Load each PDF
        loader = PyPDFLoader(str(pdf_file))
        docs = loader.load()
        all_documents.extend(docs)

        print(f"    ✅ Loaded {len(docs)} pages")

    print(f"\n📊 Total: {len(all_documents)} pages from {len(pdf_files)} PDFs")

    # Show unique sources
    sources = set(doc.metadata['source'] for doc in all_documents)
    print(f"\nSources:")
    for source in sources:
        print(f"  - {Path(source).name}")

else:
    print(f"❌ Directory not found: {pdf_directory}")

4. Loading CSV Files 📊

🔰 BEGINNER

CSVLoader converts each row of a CSV file into a separate Document.

Use cases:

  • Product catalogs
  • FAQ databases
  • Customer records
  • Any tabular data
Python 
from langchain_community.document_loaders import CSVLoader

# Load the products CSV
csv_path = "sample_data/products.csv"

if Path(csv_path).exists():
    print(f"Loading CSV: {csv_path}\n")

    # Create loader
    loader = CSVLoader(
        file_path=csv_path,
        source_column="product_name"  # Which column to use as source in metadata
    )

    # Load all rows
    documents = loader.load()

    print(f"✅ Loaded {len(documents)} products\n")

    # Inspect first 3 products
    for i, doc in enumerate(documents[:3], 1):
        print(f"{'='*70}")
        print(f"Product {i}:")
        print(f"{'='*70}")
        print(doc.page_content)
        print(f"\nSource: {doc.metadata['source']}")
        print(f"Row: {doc.metadata.get('row', 'N/A')}")
        print()

    print(f"... and {len(documents) - 3} more products")

else:
    print(f"❌ CSV not found: {csv_path}")

5. Loading JSON Files 🔧

🔰 BEGINNER

JSONLoader extracts data from JSON files using jq syntax (a query language for JSON).

Common use cases:

  • API responses
  • Configuration files
  • Structured data exports
Python 
from langchain_community.document_loaders import JSONLoader

# Load the API response JSON
json_path = "sample_data/api_response.json"

if Path(json_path).exists():
    print(f"Loading JSON: {json_path}\n")

    # Create loader
    # jq_schema tells us where to find the content in the JSON
    # .articles[] means: get all items from the 'articles' array
    loader = JSONLoader(
        file_path=json_path,
        jq_schema=".articles[]",  # Extract each article
        text_content=False  # Return full JSON for each article
    )

    # Load articles
    documents = loader.load()

    print(f"✅ Loaded {len(documents)} articles\n")

    # Inspect first article
    print("📰 First Article:")
    print(f"Content:\n{documents[0].page_content}\n")
    print(f"Metadata: {documents[0].metadata}")

else:
    print(f"❌ JSON not found: {json_path}")

🔰 BEGINNER TIP: Understanding jq Syntax

jq is like a GPS for JSON:

jq ExpressionMeaning
.Root of JSON
.articlesGet the 'articles' field
.articles[]Get all items in 'articles' array
.articles[0]Get first item in 'articles' array
.articles[].titleGet 'title' from each article

Example:

JSON 
{
  "articles": [
    {"title": "Article 1", "content": "..."},
    {"title": "Article 2", "content": "..."}
  ]
}
  • .articles[] → Returns both articles
  • .articles[].title → Returns ["Article 1", "Article 2"]

8. Batch Loading with DirectoryLoader 📂

🔰 BEGINNER

DirectoryLoader loads all files from a directory automatically.

Perfect for:

  • Loading entire document libraries
  • Processing multiple files at once
  • Building knowledge bases
Python 
from langchain_community.document_loaders import DirectoryLoader

# Load all files from sample_data directory
data_dir = "sample_data"

if Path(data_dir).exists():
    print(f"📂 Loading all text files from: {data_dir}/\n")

    # Create loader for .txt files only
    loader = DirectoryLoader(
        data_dir,
        glob="*.txt",  # Pattern to match files
        loader_cls=TextLoader,  # Use TextLoader for each file
        show_progress=True  # Show progress bar
    )

    # Load all matching files
    documents = loader.load()

    print(f"\n✅ Loaded {len(documents)} text file(s)\n")

    # Show sources
    for doc in documents:
        print(f"  - {doc.metadata['source']} ({len(doc.page_content)} chars)")

else:
    print(f"❌ Directory not found: {data_dir}")

🎓 INTERMEDIATE: Loading Multiple File Types

Python 
# Advanced: Load all files from a directory (mixed types)
# This function handles different file types intelligently

def load_all_documents(directory: str) -> list:
    """
    Load documents from multiple file formats in a directory.

    Supports: PDF, TXT, CSV, JSON, HTML
    """
    all_docs = []
    directory_path = Path(directory)

    if not directory_path.exists():
        print(f"❌ Directory not found: {directory}")
        return []

    print(f"📂 Loading from: {directory}\n")

    # Load PDFs
    pdf_files = list(directory_path.glob("*.pdf"))
    for pdf in pdf_files:
        loader = PyPDFLoader(str(pdf))
        docs = loader.load()
        all_docs.extend(docs)
        print(f"  ✅ PDF: {pdf.name} ({len(docs)} pages)")

    # Load TXT files
    txt_files = list(directory_path.glob("*.txt"))
    for txt in txt_files:
        loader = TextLoader(str(txt))
        docs = loader.load()
        all_docs.extend(docs)
        print(f"  ✅ TXT: {txt.name}")

    # Load CSV files
    csv_files = list(directory_path.glob("*.csv"))
    for csv in csv_files:
        loader = CSVLoader(str(csv))
        docs = loader.load()
        all_docs.extend(docs)
        print(f"  ✅ CSV: {csv.name} ({len(docs)} rows)")

    # Load JSON files
    json_files = list(directory_path.glob("*.json"))
    for json_file in json_files:
        try:
            loader = JSONLoader(
                str(json_file),
                jq_schema=".",
                text_content=False
            )
            docs = loader.load()
            all_docs.extend(docs)
            print(f"  ✅ JSON: {json_file.name}")
        except Exception as e:
            print(f"  ⚠️ JSON: {json_file.name} (error: {str(e)[:50]}...)")

    print(f"\n📊 Total: {len(all_docs)} documents loaded")
    return all_docs

# Test the function
if Path("sample_data").exists():
    all_documents = load_all_documents("sample_data")

    # Show summary
    print(f"\n📈 Summary:")
    sources = [doc.metadata['source'] for doc in all_documents]
    print(f"   Files loaded: {len(set(sources))}")
    print(f"   Total documents: {len(all_documents)}")

⚡ Quick Recap

  • A LangChain Document is the universal unit passed downstream: a page_content string plus a metadata dict. Every loader, regardless of source format, normalizes its input into these objects.
  • All loaders share one interface — .load() returns the full list[Document] in memory, while .lazy_load() returns a generator that yields one Document at a time to keep memory flat on large inputs.
  • Granularity is loader-specific: PyPDFLoader emits one Document per page, CSVLoader one per row, TextLoader one per file, and JSONLoader a count that depends entirely on the jq query.
  • JSONLoader uses a jq_schema to select which nodes become Documents (.articles[] = each array element); text_content=True requires the selected node to be a scalar string, False serializes the whole node.
  • WebBaseLoader and UnstructuredHTMLLoader only extract static server-rendered HTML — JavaScript-rendered content needs a browser-driven loader such as Playwright or Selenium.
  • DirectoryLoader batch-loads files matching a glob pattern using a specified loader_cls; mixed-format directories are handled by dispatching per extension in your own code.
  • Defensive loading is standard practice: guard with Path(...).exists() and wrap .load() in try/except, since loaders raise on missing files, missing optional dependencies (jq, unstructured), and malformed content.
  • Metadata (source, page, row) is auto-populated and can be augmented post-load — it is what enables later filtering and citation in a retrieval pipeline.

🎯 Interview Prep

Q1. What is a LangChain Document and why is it central to a RAG pipeline?

A Document is a simple container with two fields: page_content, the text itself, and metadata, a dictionary describing where that text came from. Its value is that it decouples the ingestion source from the rest of the pipeline — a PDF, a CSV row, and a scraped web page all arrive downstream as the same object. That means my splitter, embedder, and retriever never need to know the original format. The metadata also travels with the text, so I can filter and cite sources at retrieval time.

Follow-up: Interviewers may ask why metadata matters — the answer is provenance and filtering: it lets you do metadata-scoped retrieval (e.g. only this year's docs) and return citations, which are essential for trustworthy RAG.

Q2. When would you use lazy_load() instead of load()?

I use load() for small or moderate inputs where holding everything in memory is fine, because it's simpler and returns a list I can index and slice. I switch to lazy_load() for large files — a several-hundred-page PDF or a huge CSV — because it yields Documents one at a time as a generator, keeping peak memory proportional to a single item rather than the whole corpus. It also lets me start processing (splitting, embedding) before the entire file is read. The tradeoff is I lose random access and a known length up front.

Gotcha: lazy loading only helps if the downstream stage also streams — if you immediately do list(loader.lazy_load()) you've reintroduced the full-memory footprint you were trying to avoid.

Q3. Explain how JSONLoader's jq_schema works and the role of text_content.

The jq_schema is a jq query that selects which nodes in the JSON become Documents — .articles[] turns each element of the articles array into its own Document, while .articles[].content drills down to just the content string of each. The text_content flag controls how the selected node is interpreted: True expects a scalar string and uses it directly as page_content, whereas False serializes the whole selected object into the content. Choosing wrong is a common error — pointing at an object with text_content=True raises. So I match the flag to whether my query lands on a string or a structure.

Gotcha: JSONLoader depends on the native jq package, which can be awkward to install on Windows — a frequent reason the loader fails in fresh environments.

Q4. Why can't WebBaseLoader scrape every website, and what do you do about it?

WebBaseLoader fetches the raw HTML the server returns and parses it — it does not run JavaScript. So for modern single-page apps where the content is rendered client-side, the fetched HTML is largely empty and the loader extracts little or nothing. When I hit that, I move to a browser-based loader like the Playwright or Selenium loaders, which spin up a real browser, execute the JS, and then extract the rendered DOM. The cost is much slower loads and heavier dependencies, so I only reach for them when static extraction genuinely fails.

Follow-up: Be ready to mention rate-limiting, robots.txt, and setting a proper User-Agent — production scraping needs politeness and error handling, not just the loader call.

Q5. How does DirectoryLoader handle a folder containing mixed file types?

DirectoryLoader takes a glob pattern and a single loader_cls, so out of the box it applies one loader to all files matching the pattern — for example *.txt with TextLoader. It doesn't auto-select a loader per extension. For genuinely mixed folders I either run it multiple times with different globs and loader classes, or write a small dispatcher that iterates the directory, inspects each extension, and instantiates the right loader — which is exactly what the load_all_documents function in this notebook does.

Gotcha: a single failing file can abort the whole batch, so wrap per-file loading in try/except and log failures — that's why the notebook's JSON branch catches exceptions individually.

Q6. What granularity does each common loader produce, and why does it matter?

PyPDFLoader produces one Document per page, CSVLoader one per row, TextLoader one per file, and JSONLoader a variable count driven by the jq query. This matters because granularity sets the natural retrieval unit before any splitting: CSV rows are already nicely self-contained chunks, whereas a PDF page may still be too large and need further splitting, and a whole text file is almost always too big to embed as one vector. Knowing the starting granularity tells me how aggressive my text splitter needs to be next.

Follow-up: A common probe is "is one-Document-per-page always ideal for PDFs?" — no, semantic content often spans page boundaries, so page-level Documents plus a recursive splitter usually beats relying on pages alone.

Q7. What are the key failure modes when loading documents, and how do you guard against them?

The main ones are missing files, missing optional dependencies, and malformed or unreadable content. I guard file existence with Path(path).exists() before constructing the loader, and I wrap .load() in try/except catching both FileNotFoundError and a general Exception so one bad file doesn't crash a batch. For dependency issues — like jq for JSONLoader or unstructured for HTML/Markdown — I catch ImportError and fall back to a simpler loader such as TextLoader. Logging what failed and why keeps the ingestion pipeline observable.

Gotcha: encoding is an easily-missed one — TextLoader can raise on non-UTF-8 files, so passing an explicit encoding or using autodetection avoids silent corruption or hard failures.
