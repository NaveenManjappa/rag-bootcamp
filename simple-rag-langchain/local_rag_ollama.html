Local Offline RAG with Ollama

Overview

This notebook demonstrates building a completely offline RAG (Retrieval-Augmented Generation) system using Ollama for local LLMs and embeddings.

🚀 Benefits of Local RAG:

  • 100% Offline: No internet required after setup
  • Privacy First: Your documents never leave your machine
  • No API Costs: Free to run unlimited queries
  • Fast: No network latency
  • Full Control: Customize models and parameters

📋 Architecture:

PDF Documents → Load → Split → Local Embeddings (Ollama) → ChromaDB
                                                                  ↓
User Query → Retrieve Similar Chunks → Local LLM (Ollama) → Answer

🛠️ Components:

  • Document Loader: PyPDFLoader
  • Text Splitter: RecursiveCharacterTextSplitter
  • Embeddings: Ollama with nomic-embed-text (or embeddinggemma)
  • Vector Store: ChromaDB (persistent, local)
  • LLM: Ollama with gemma3:1b
  • Chain: LangChain Expression Language (LCEL)

# Initialize text splitter
text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=1024,        # Characters per chunk
    chunk_overlap=128,      # Overlap to maintain context
    length_function=len,
    separators=["\n\n", "\n", " ", ""]  # Split on paragraphs, then lines, etc.
)

# Split documents
chunks = text_splitter.split_documents(documents)

# Display results
avg_chunk_size = sum(len(chunk.page_content) for chunk in chunks) / len(chunks)

11. Build RAG Chain (LangChain Expression Language)

Combine retrieval and generation into a single pipeline using LCEL.

code · cell-30
Python 
# Define prompt template
system_prompt = (
    "You are a helpful assistant for question-answering tasks. "
    "Use the following pieces of retrieved context to answer the question. "
    "If you don't know the answer based on the context, say that you don't know. "
    "Keep the answer concise and accurate.\n\n"
    "Context: {context}\n\n"
    "Question: {question}"
)

prompt = ChatPromptTemplate.from_template(system_prompt)

# Helper function to format documents
def format_docs(docs):
    """Format retrieved documents into a single string."""
    return "\n\n".join(doc.page_content for doc in docs)

# Build RAG chain using LCEL
rag_chain = (
    {
        "context": retriever | format_docs,  # Retrieve and format docs
        "question": RunnablePassthrough()      # Pass through the question
    }
    | prompt           # Format with prompt template
    | llm              # Generate answer with local LLM
    | StrOutputParser() # Parse output to string
)

print("✓ RAG chain created successfully using LCEL!")
print("\nRAG Pipeline Flow:")
print("  1. User provides a query")
print("  2. Retriever finds top 4 relevant chunks (local ChromaDB)")
print("  3. Chunks are formatted as context")
print("  4. Context + question formatted with prompt template")
print("  5. Local LLM (gemma3:1b) generates answer")
print("  6. Answer parsed and returned")
print("\n🔒 Everything runs locally on your machine!")

Quick Recap

  • Local-first RAG: The entire pipeline — embeddings, vector store, and generation — runs on-device via Ollama, so no data leaves the machine and there are no per-query API costs after the models are pulled.
  • Ingestion flow: PyPDFLoader loads the PDF page-by-page, then RecursiveCharacterTextSplitter breaks it into chunks of ~1024 chars with 128-char overlap to preserve context across boundaries.
  • Embeddings: nomic-embed-text (274 MB) converts each chunk into a 768-dimensional vector locally; embeddinggemma is a drop-in alternative optimized for Gemma LLMs.
  • Vector store: ChromaDB persists the embeddings to disk at ./chroma_db under a named collection, so the index can be reloaded later without re-embedding.
  • Retrieval: The retriever uses cosine similarity to fetch the top k=4 most relevant chunks for a query, which become the grounding context.
  • Generation: A local ChatOllama model with temperature=0 produces deterministic answers; larger models (llama3.2, deepseek-r1) trade speed for quality.
  • LCEL composition: The chain pipes a dict of {context: retriever | format_docs, question: RunnablePassthrough()} → prompt template → LLM → StrOutputParser, giving a clean, composable, single-call pipeline.
  • Answer quality: Output is only as good as the retrieved context and the LLM — small local models limit generation quality even when retrieval is sound, so model choice and chunking both matter.

Interview Prep

1. Why would you build a fully local RAG system instead of using a cloud API?

I'd choose local RAG when privacy, cost, or offline availability are priorities. Because embeddings, the vector store, and generation all run on-device via Ollama, sensitive documents never leave the machine, there are no per-token API charges, and there's no network latency or dependency. The trade-off is that local models are typically smaller and less capable than frontier cloud models, and you're responsible for your own hardware and scaling.

Gotcha: Interviewers may push on quality — the honest answer is that small local models can produce weaker or hallucinated answers, so local RAG is best when the retrieval context does most of the heavy lifting and the generation task is relatively simple.

2. What does chunk size and chunk overlap control, and how do you tune them?

Chunk size sets how much text each vector represents; overlap repeats characters across adjacent chunks so context isn't lost at boundaries. Smaller chunks give more precise retrieval but risk fragmenting ideas, while larger chunks preserve context but dilute the embedding and can retrieve irrelevant text. Here I used 1024 characters with 128 overlap as a balanced default, and I'd tune by evaluating retrieval quality on representative queries.

Gotcha: Overlap that's too large wastes storage and can return near-duplicate chunks; a good follow-up is to mention token-based splitting instead of character-based when the embedding model has a strict token limit.

3. How does the retriever decide which chunks are relevant?

The query is embedded into the same vector space as the chunks, and the retriever performs a similarity search — here cosine similarity — returning the top k nearest vectors. I set k=4, meaning the four closest chunks become the context passed to the LLM. The key assumption is that semantic closeness in embedding space corresponds to relevance to the question.

Gotcha: Pure similarity search can miss relevant chunks that are worded differently; a strong follow-up is mentioning hybrid search (dense + keyword/BM25) or re-ranking to improve recall and precision.

4. Walk me through the LCEL chain in this notebook.

The chain starts with a dictionary where context is retriever | format_docs (retrieve chunks then join them into one string) and question is RunnablePassthrough() which forwards the raw query unchanged. That dict feeds a ChatPromptTemplate, whose output goes to the ChatOllama LLM, and finally StrOutputParser converts the message object to a plain string. The pipe operator makes each stage composable and independently testable.

Gotcha: A common probe is why RunnablePassthrough is needed — it's because the retriever consumes the query for context, but the prompt template also needs the original question verbatim, so it must be forwarded in parallel.

5. Why persist the vector store, and what does ChromaDB give you here?

Embedding documents is the most expensive step, so persisting to disk means I can reload the index instantly instead of re-embedding on every run. ChromaDB provides a simple, local, persistent vector database that stores vectors under a named collection at a directory I specify. This makes the system restart-friendly and lets me separate the one-time ingestion step from repeated querying.

Gotcha: A subtle pitfall — if you change the embedding model, the persisted vectors are incompatible with new queries, so you must re-embed; interviewers like to see that you know the embedding model and the store are tightly coupled.

6. What role does temperature play, and why set it to 0 for RAG?

Temperature controls randomness in generation — lower values make the model more focused and deterministic, higher values more creative and varied. For RAG question-answering I set it to 0 because I want factual, reproducible answers grounded strictly in the retrieved context, not creative embellishment. If the task were brainstorming or drafting, I'd raise it to 0.3–0.7.

Gotcha: Temperature 0 reduces but does not eliminate hallucination — the model can still invent facts if the prompt doesn't strongly constrain it, which is why the system prompt explicitly tells it to say "I don't know" when the context lacks the answer.

7. The prompt instructs the model to say "I don't know" if the answer isn't in the context. Why does this matter?

Grounding instructions reduce hallucination by giving the model explicit permission to abstain rather than fabricate. In RAG, the whole point is that answers should be traceable to retrieved sources, so an "I don't know" is a correct and safe response when retrieval fails. It also makes failures debuggable — an empty or refused answer signals a retrieval problem rather than masking it with a confident wrong answer.

Gotcha: Small models often ignore such instructions or answer anyway; a good follow-up is to mention returning source documents alongside answers so users can verify grounding themselves, as this notebook does.

8. How would you improve answer quality if this local RAG gave weak results?

First I'd separate retrieval quality from generation quality: inspect the retrieved chunks to confirm the right context is being fetched, tuning chunk size, k, or adding re-ranking if not. If retrieval is good but answers are poor, I'd upgrade to a larger local model like llama3.2 or deepseek-r1, or refine the prompt. I'd also consider cleaning the source PDF extraction, since noisy chunks degrade both retrieval and generation.

Gotcha: The classic mistake is blaming the LLM first — the interviewer wants to hear that in RAG most quality problems originate in ingestion and retrieval, so you should diagnose those before swapping models.
